1. What this tool does

This importer helps you move data from a Firebird .gdb database into an existing SQL Server database. It is mainly intended for controlled Ortems data migration, data validation, and demo-data transfer scenarios.

Use it to

  • Connect to a source Firebird .gdb file.
  • Connect to a target SQL Server database.
  • Select which tables to import.
  • Run an import with progress, counters, and a log file.
  • Export Firebird tables as SQL Server-compatible scripts.

Do not use it to

  • Create the SQL Server database structure from scratch.
  • Fix an unknown or corrupted production database without a backup.
  • Open Ortems against a partial database after an aborted import.
Key point: the SQL Server database structure must already exist before the import starts.

2. Before you start

Most avoidable failures happen before the first row is imported. Check these items first.

Source side

  • You can access the Firebird .gdb file.
  • The Firebird ODBC driver is installed.
  • You know the Firebird server, port, user, and password.

Target side

  • The SQL Server database already exists.
  • The Ortems objects/tables already exist in the target database.
  • You have permission to insert data and, when selected, clear target tables.
  • You have a recent backup before any reload.

3. Quick import flow

Use this sequence for the normal import process. It is intentionally simple: connect, load tables, select, run, review.

Select the source file
Click Browse... and choose the Firebird .gdb file.
Test the Firebird connection
Confirm the Firebird server, port, user, and password. Click Test Firebird Connection.
Enter the SQL Server target
Fill in the SQL Server name, database, authentication method, and target schema. The default schema is usually dbo.
Test the SQL Server connection
Click Test SQL Server Connection. Do not continue until it succeeds.
Load the table list
Click 1 - Load Table List. The source tables appear in the table list.
Select the tables
Click 2 - Select All for a full migration, or check only the tables you want.
Review the options
For most Ortems imports, keep the default safe options unless you have a reason to change them.
Run the import
Click 3 - Run Import and monitor the progress bar, status line, counters, and live log.
Read the final message
At the end, review the result message and open the log if there are warnings, skipped rows, failed tables, or errors.

6. Import options

These options change how the import behaves. When in doubt, use a test database first.

OptionPlain-English meaningRecommended use
Clear selected tables before loadDeletes existing data from the selected target tables before importing.Use for a clean reload. Back up the target first.
Skip duplicate PK rows in append modeIf a row already exists, the importer can skip it instead of stopping immediately.Useful when appending data into a database that may already contain some records.
Order tables by SQL Server FK dependenciesLoads parent tables before dependent tables when the target database metadata allows it.Keep enabled for Ortems imports.
Convert empty strings to NULL when target column is nullableTreats blank values as empty database values when the target column allows it.Usually safe and helpful for cleaner data.
Use safe fallback/default for NULL values in NOT NULL columnsUses safe fallback values when the target field requires a value but the source is blank.Use when loading Ortems datasets that may contain blanks in mandatory fields. Review the log.
Temporarily disable target table triggers during clear/loadPrevents target-side automation from running while bulk data is loaded.Useful for full reloads where triggers may create side effects. Verify the final log.
Verbose modeWrites detailed diagnostic information to the log file.Turn on when troubleshooting. Leave off for normal faster runs.
Log BatchControls how often row progress is refreshed during long operations.Keep the default 500 unless support asks for a different value.
Practical default: for most controlled Ortems migrations, keep foreign-key ordering enabled and review the log at the end, even when the import succeeds.

7. Export SQL scripts

The export feature creates SQL Server-compatible text scripts from the Firebird source tables. It does not import data into SQL Server; it only writes files.

Select the source GDB
Choose the Firebird file and test the Firebird connection.
Load and select tables
Load the table list and check the tables you want to export. If no tables are checked, the tool may export all source user tables.
Click Export SQL Scripts
The application creates an export folder under .\Export\.
Use the master script if needed
The generated folder includes a 00_RUN_ALL.sql master file plus one script per table.
Review the export log
The export log is saved under .\log\export_YYYYMMDD_HHMMSS.log.
Good use case: use export when you need a readable transfer package, a reviewable SQL backup, or scripts that can be executed manually later.

8. Progress and logs

What you see on screen

  • The progress bar shows overall advancement.
  • The progress label shows the active table and row counters when available.
  • The footer shows the current status, errors, and warnings.
  • The live log shows key messages while the process is running.

Where files are saved

  • Import logs: .\log\import_YYYYMMDD_HHMMSS.log
  • Export logs: .\log\export_YYYYMMDD_HHMMSS.log
  • SQL script exports: .\Export\[GDBName]_YYYYMMDD_HHMMSS\
Keep failed-run logs. They are the fastest way to understand connection problems, missing tables, skipped rows, duplicate rows, and failed imports.

10. Preparing a blank SQL Server database

The importer expects the SQL Server tables and database objects to already exist. If the database is blank, create the Ortems objects first.

Open Ortems Environment Manager
Create or select the environment that points to the target SQL Server database.
Enter the SQL Server connection
Confirm server, database, authentication, user/password if required, and schema.
Test the connection
Make sure Environment Manager can reach the target database.
Create the database objects
If Environment Manager indicates the database is empty, open the object creation window and create the objects.
Return to the importer
Test the SQL Server connection in this tool, load the table list, select the tables, and run the import.
Do not run the import against a blank database. If object creation fails, ask a database administrator to verify permissions.

11. Troubleshooting

SymptomMost likely causeTry this first
Firebird connection failsWrong file path, server, port, credentials, or missing driver.Confirm the .gdb path, Firebird service, ODBC driver, user, and password.
SQL Server connection failsWrong server/database, authentication problem, network issue, or missing permissions.Retype the server and database. Test Windows vs SQL Server authentication as appropriate.
No tables appearThe source connection did not succeed or the selected source does not contain user tables.Test Firebird connection again, then click 1 - Load Table List.
Import fails immediatelyMissing source file, no selected tables, or target connection failure.Read the final message and check the log path shown in the result.
Duplicate-row errorsThe target database already contains matching rows.Use a clean database reload or enable duplicate skipping only when append mode is intentional.
Foreign-key errorsRequired parent data is missing or selected tables are incomplete.Keep FK ordering enabled and include the related parent tables in the import.
Warnings after successThe importer adjusted, skipped, or corrected data during the run.Review the warning lines in the log before relying on the database.
Ortems does not open after importThe target data may be incomplete or inconsistent.Do not keep using that database. Review the latest import log and the Ortems startup message together.

12. Safe operating practices

Use a test target first

Run the import into a test SQL Server database before touching any shared or production-like environment.

Back up before reload

Always back up the target database before using Clear selected tables before load.

Do not ignore warnings

A completed import can still contain warnings. Read the log before opening Ortems or sharing the result.

Avoid partial databases

If an import is aborted or fails, treat the target as incomplete until it is corrected and reloaded.

Keep evidence

When reporting a problem, include the log file, selected options, source GDB name, target database name, and screenshots of any final message.

Change one thing at a time

When troubleshooting, avoid changing many options at once. That makes it much harder to identify the real cause.

Simple rule: successful connection tests plus a clean final log are the best signs that the dataset is ready for validation.
